# 기능 개선 요건 정의서 (PRD): 워크스페이스 고도화 및 관리자 기능 확장

## 1. 개요 (Overview)
본 문서는 LogiVOC 시스템의 사용자 경험을 개선하고, 관리자 및 데이터 등록 워크플로우를 고도화하기 위한 요구사항을 정의합니다. 시스템 관리 기능의 확장, 워크스페이스 탭의 역할 재정의, UI 레이아웃 개선, 그리고 AI 파이프라인의 유연성을 높이는 커스텀 프롬프트 및 사용자 정의 카테고리 추가 기능이 포함됩니다.

## 2. 변경 전/후 시스템 비교 (As-Is vs To-Be)

### 2.1 탭(Tab) 구조 변경
*   **As-Is**: `Workspace` (문서 업로드 및 검토) / `Search` (검색 중심) / `System Admin` (카테고리 관리 중심)
*   **To-Be**: `Workspace` (검색 중심) / `데이터 등록` (문서 업로드 및 검토) / `System Admin` (전체 DB 조회 등 통합 관리)

## 3. 핵심 요구사항 (Core Requirements)

### 3.1 시스템 관리자(System Admin) 페이지 확장
*   **현재 상태**: 신규 카테고리(new category) 추가 등 제한적인 관리 기능만 존재.
*   **요구사항**: 
    - DB에 저장된 전체 데이터(문서, 지식, 사용자, 카테고리 등)를 종합적으로 조회할 수 있는 '전체 데이터 조회' 대시보드 및 리스트 뷰 기능 추가.
    - 데이터 상태(DRAFT, PUBLISHED)와 관계없이 모든 시스템 데이터에 대한 관리자 권한의 조회, 필터링, 수정, 삭제 인터페이스 제공.

### 3.2 워크스페이스(Workspace) 탭 및 데이터 등록 탭 분리
*   **요구사항**:
    - **워크스페이스 탭**: 기존 Search/Dashboard 페이지의 역할을 흡수하여, 통합 지식 검색(Search) 및 결과 조회 중심의 메인 페이지로 변경 및 롤백.
    - **데이터 등록(Data Registration) 탭 신설**: 기존 워크스페이스 탭에 있던 '문서 업로드', 'AI 파이프라인 시작', 'DRAFT 문서 검토 및 승인/반려' 기능을 새로운 탭으로 완전히 분리하여 이동.

### 3.3 데이터 등록/뷰 화면 UI 구조 변경
*   **현재 상태**: 추출된 데이터 영역과 원본 문서 뷰어 영역의 구분이 최적화되지 않음.
*   **요구사항**: 
    - 화면의 구조를 변경하여 콘텐츠(Content) 영역(AI가 추출한 텍스트, 요약, 메타데이터 등)을 원본 문서 뷰어(Original Document Viewer) 영역보다 시각적으로 더 넓게(크게) 보여주도록 UI 레이아웃 비율(예: 7:3 등) 조정.
    - 사용자가 텍스트를 검토하고 편집하는 주요 작업의 편의성 극대화.

### 3.4 AI 파이프라인 커스텀 프롬프트 (Ad-hoc Custom Prompt)
*   **요구사항**: 
    - 데이터 등록 탭에서 문서 업로드 시, AI 파이프라인을 시작하기 전에 사용자가 원하는 '추출 및 정리 방식'을 직접 입력할 수 있는 **커스텀 프롬프트 입력란** 제공.
    - 입력된 커스텀 프롬프트는 LangGraph 기반 AI 파이프라인의 처리 단계에 Context로 동적 주입되어, 문서 단위별 사용자 맞춤형 데이터화(추출/요약)가 수행되도록 반영.

### 3.5 사용자 정의 카테고리 (Custom Category)
*   **요구사항**: 
    - 기존 시스템 관리자 화면에서만 가능했던 온톨로지 카테고리 제어를 확장하여, 사용자가 데이터 등록 시에 직접 **새로운 카테고리를 추가 및 작성**할 수 있는 기능 제공.
    - 사용자가 입력한 신규 카테고리는 계층 구조(`Service > Module > Architecture`)에 맞게 동적으로 시스템에 등록되어 즉시 활용 가능해야 함.

### 3.6 보안 요구사항 (Security Requirements)
*   **요구사항**: 신규 커스텀 프롬프트 주입 및 파일 업로드(Celery 워커 연동) 기능에서 발생할 수 있는 보안 취약점을 방어하기 위해 다음 정책을 적용.
    - **Prompt Injection 방어**:
        - **프롬프트 입력 길이 제한**: 사용자가 입력할 수 있는 커스텀 프롬프트의 최대 길이를 제한 (예: 500자 이내).
        - **특수 문자 및 지시어 필터링**: 프롬프트 인젝션을 유도하는 특수 기호나 시스템 탈옥(Jailbreak) 관련 키워드 차단 및 무효화 로직 적용.
        - **시스템 프롬프트 격리**: 커스텀 프롬프트가 시스템 메인 지시어를 덮어쓰지 못하도록, SystemMessage와 HumanMessage(Context)를 엄격히 분리하여 주입.
    - **DoS(서비스 거부) 공격 방지**:
        - **파일 업로드 크기 제한**: 대용량 파일로 인한 자원 고갈을 막기 위해 파일당 최대 크기 및 동시 업로드 개수 제한.
        - **Rate Limiting**: 단시간 내 과도한 파일 업로드 및 파이프라인 실행 요청을 막기 위한 API Rate Limiting 적용.
        - **Celery 워커 타임아웃(Timeout) 제한**: 무한 루프 및 과도한 처리 대기를 방지하기 위해 Task 처리 최대 시간 제한(Soft/Hard Time Limit) 설정.

## 4. 연관 문서 및 아키텍처 영향도 (Impact Analysis)
*   **프론트엔드**: 상단 내비게이션 라우팅 탭 추가 및 명칭 변경, 분리된 화면에 맞춘 레이아웃 컴포넌트 재구성, UI 스플릿 비율 조정, 프롬프트 및 사용자 카테고리 입력 폼 구현.
*   **백엔드/API**: 파이프라인 실행 API에 커스텀 프롬프트 파라미터 추가, LangGraph Node 프롬프트 동적 주입 로직 구현, 사용자 카테고리 동적 생성 API 추가, 어드민 권한의 전체 데이터(DB) 조회용 신규 API 구현.
*   **DB 모델/스키마**: 신규 기능에 맞춰 동적 카테고리 관리에 필요한 테이블 속성 점검 및 필요시 마이그레이션(Alembic) 반영.

## 5. 단계별 후속 작업 (Next Steps)
본 문서(Phase 1) 완료 후 다음 단계의 에이전트 작업이 필요합니다:
1.  **Phase 2 (설계)**: `architect`와 `ui_ux_designer`를 호출하여 새로운 탭 구조와 화면 레이아웃, 프롬프트 주입을 위한 API 변경 스펙, 시스템 어드민 DB 조회 스펙에 대한 문서(ADR, API Spec, UI Guide) 작성을 위임합니다.
2.  **Phase 3 (구현)**: 설계 문서를 바탕으로 `frontend_dev`와 `backend_dev`가 기능 개발을 수행합니다.
